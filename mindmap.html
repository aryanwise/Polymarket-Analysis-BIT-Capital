1 Polymarket Extraction
  • Hits Polymarket API with pagination
  • Default max events: 3 000 — configurable via --max-events flag
  • Page size: 500 per request
  • Offset starts at 0, increments per batch
2 Filtering & Enrichment
Stage 1 — Rule-Based Filtering
  • Zero volume & missing prices — Drop markets with no liquidity or absent price data
  • Expired markets — Remove past end-date events
  • Fully resolved markets — Cut already settled outcomes
  • Near-certain ones — Skip YES < 4% or > 96% (no actionable uncertainty)
  • Irrelevant tags — Filter by non-target categories
  • Dedup by event — One market per event using signal quality score: uncertainty × 0.65 + volume × 0.35
  • Minimum volume $5k — Low liquidity = unreliable probability signal
Stage 2 — Three-Pass LLM Pipeline

Pass 1 — Mistral gate

Fast binary SIGNAL / NOISE on every market. Fail-open — on parse error defaults to SIGNAL so nothing slips through.

Mistral Small · temp=0.0 · batch=20

Pass 2 — Groq classifier

Maps each market to specific tickers from the 14 holdings. Fail-closed — on parse error returns NOISE, nothing wrong written to DB.

Llama 4 Scout · temp=0.0 · batch=10

Pass B — Groq enrichment

Fills sentiment (Bullish/Bearish/Neutral), impact_score 1–10, and reasoning with a specific dollar figure for every signal.

Llama 4 Scout · temp=0.1 · batch=15
⚡ Incremental filter (Step 2.5): New markets (~7 per run) → sent to LLM. Known markets (~70 per run) → yes_price and volume refreshed via PATCH only. Results in 95% reduction in LLM calls on mature runs.
DB Supabase Data Layer
  • signals — core table · 280+ rows · market_id, ticker, yes_price, sentiment, impact_score, reasoning
  • signal_feed — VIEW that joins signals with stocks (adds company name, thesis, sector)
  • reports — generated markdown reports with metadata (model used, tickers, signal count)
  • report_signals — junction table linking each report to the signals it referenced
  • signal_outcomes — backtesting results, first data June 17 2026
  • stock_prices — price snapshots from Yahoo Finance for historical lookup
  • Upsert logic — ON CONFLICT (market_id, ticker) DO UPDATE keeps prices fresh without duplicates
3 Report Generation & Live Dashboard
Report Flow
  • Fetch top 30 signals from signal_feed ordered by impact_score
  • Price context — Yahoo Finance injects live price, 1D/5D change, 52-week range position, FMP analyst targets for all 14 holdings into the prompt
  • Crowd vs Reality — DDGS news search on top 3 signals · Groq computes news sentiment vs Polymarket probability · HIGH divergence = information gap = alpha
  • Gemini 2.5 Flash generates the report · temp=0.1 · 16k tokens · Groq Llama 4 Scout as fallback
  • Crowd vs Reality section injected via post-processing (not through LLM) to preserve content
  • Report saved to DB · signals linked via report_signals junction table
Edge Case Handling

Fail Open — Mistral (Pass 1)

On parse error → defaults to SIGNAL. Missing a real signal costs more than one extra Groq call. False positives are cheap.

batch=20 · temp=0.0

Fail Closed — Groq (Pass 2)

On parse error → returns NOISE. At classification stage precision matters — nothing wrong should be written to DB. False negatives are acceptable.

batch=10 · temp=0.0
Real-time Prices — prices.py
  • Two consumers: report generator (dict injected into LLM prompt) and Streamlit Holdings tab (list for price cards)
  • Yahoo Finance: current price · 1D change · 5D change · 52-week high/low · range position label
  • FMP (optional): analyst price targets and upside % for all 14 holdings
  • Saved to stock_prices table each run for historical backtesting lookup
Backtesting — backtest.py
  • First data: June 17 2026 — Fed meeting markets are first to expire
  • Finds signals where end_date < today · checks YES price ≥ 0.92 = resolved YES, ≤ 0.08 = resolved NO
  • Fetches stock price at signal creation vs expiry via yfinance · checks if sentiment direction matched actual move
  • Key metric: do impact_score 8–10 signals have higher accuracy than score 3–5? If yes, the scoring model generates real alpha
  • Results written to signal_outcomes table
+ Extra Features
Dig Deeper — dig_deeper_analysis.py

Analyst clicks "Compare vs news" on any signal. Runs DDGS multi-query news search, extracts headlines with article bodies, calls Groq for Agreement/Conflict analysis and directional call. Cached 6 hours in deep_dives table to avoid redundant searches.

Explore Market — explore_polymarket_news.py

The Explore Market tab embeds an ad-hoc market explorer powered by pipeline/explore_polymarket_news.py. It lets you analyse any Polymarket event on demand — not just the ones surfaced by the automated pipeline — and instantly see how it relates to the BIT Capital portfolio.
